## 简介
本仓库采用"策略模式"对数据库访问层进行抽象：在浏览器环境下使用 Dexie（IndexedDB），在 Electron 环境下通过 IPC 调用主进程的 SQLite（sql.js）。**新增的HTTP后端**使得纯浏览器环境也能访问Electron的SQLite数据库，通过REST API桥接两者。前端统一通过 facade 暴露的函数与默认导出代理访问数据，业务代码无需感知底层实现差异。该设计使得同一套 Zustand store、服务与页面可在三种运行时中无缝切换。

## 项目结构
围绕数据库抽象的关键路径如下：
- 启动阶段：渲染进程初始化时按优先级选择并打开后端；Electron 主进程初始化 SQLite 并注册 IPC 和 HTTP API。
- 策略分发：facade 根据运行环境创建具体后端实例，并将所有方法转发到对应实现。
- 查询层：Dexie 直接操作 IndexedDB；Electron 侧通过 IPC 将请求路由到 SQL 查询层；HTTP 后端通过 REST API 访问相同查询层。
- 持久化：Dexie 由浏览器管理；SQLite 通过 sql.js 内存库 + 定时落盘。

## 核心组件
- 数据库门面（策略入口）
  - 负责三层次环境检测、后端选择、统一 API 转发与兼容旧版直连访问。
- Dexie 后端（浏览器）
  - 基于 Dexie 的表结构与索引，提供与 Electron 后端一致的接口。
- Electron 后端（桌面）
  - 通过 window.electronAPI.db.* 与 fs.* 进行 IPC 通信，并对返回值做归一化处理。
- **HTTP 后端（浏览器+Electron桥接）**
  - 通过 fetch API 调用 Electron API 服务器的 REST 端点，提供完整的浏览器-SQLite桥接。
- SQLite 主进程层
  - 使用 sql.js 加载/保存数据库，执行 schema 初始化，并提供 debounced 落盘。
- SQL 查询层
  - 封装所有 CRUD 与统计查询，映射为与 Dexie 后端一致的数据形态。
- IPC 处理器
  - 将渲染进程请求映射到 queries.cjs 的具体函数。
- **API 服务器**
  - 内嵌于 Electron 主进程的 HTTP 服务器，提供 REST API 访问 SQLite 数据。
- **Vite 开发代理**
  - 开发环境下将 /api/db 请求代理到 Electron API 服务器。

## 性能考量
- 查询优化
  - Dexie 使用 orderBy/where 与复合索引提升筛选效率；Electron 侧使用 SQL 索引与聚合查询。
- 写入批量化
  - SQLite 侧采用 300ms 防抖落盘，降低频繁 IO 开销。
- 网络与内存
  - Electron 模式下图片以 Blob 形式在渲染进程缓存，刷新后重建 URL，避免重复下载。
  - HTTP 模式下通过流式传输大文件，减少内存占用。
- 兼容性折衷
  - Electron 端在 getImages 中对状态过滤进行 JS 层补齐，牺牲少量性能换取行为一致性。
  - HTTP 后端在网络延迟下可能影响用户体验，但提供了更大的灵活性。

## 故障排查指南
- 常见问题定位
  - 启动失败：检查 main.jsx 的 initDatabase 是否成功；Electron 主进程是否完成 SQLite 初始化与 IPC 注册。
  - 数据不一致：确认 Electron 后端对返回值做了归一化（如统计字段 hot→hotZone）。
  - 图片缺失：检查 electron-backend 的 readBlob 与 fs.readImage/readThumbnail 是否正常返回 Blob。
  - 文件夹删除异常：确认递归删除逻辑与图片解绑顺序。
  - **HTTP连接失败**：检查 API 服务器是否在正确端口启动，Vite代理配置是否正确。
  - **二进制文件传输错误**：确认 Content-Type 设置和文件大小限制。
- 建议日志位置
  - 门面与环境选择：database.js
  - Dexie 后端：dexie-backend.js
  - Electron 后端：electron-backend.js
  - **HTTP 后端**：http-backend.js
  - **API 服务器**：api-server.cjs
  - **Vite 代理**：api-proxy.js
  - SQLite 初始化与落盘：index.cjs
  - SQL 查询错误：queries.cjs
  - IPC 映射：ipc-handlers.cjs

## 结论
通过增强的策略模式，本项目实现了跨环境的数据库抽象：浏览器与 Electron 共享同一套 API，上层业务无需关心底层差异。**新增的HTTP后端**进一步扩展了应用场景，使得纯浏览器环境也能访问Electron的SQLite数据库。Dexie、Electron IPC 和 HTTP REST 三种后端均遵循统一的接口契约，并通过 IPC、REST API 与查询层完成数据持久化。该设计具备良好的可扩展性与可维护性，便于未来引入更多后端实现，同时保持了向后兼容性。
